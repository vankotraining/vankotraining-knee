# Stabilizacni checkpoint - 2026-07-06

Repo: `vankotraining/vankotraining-knee`  
Domena: `knee.vankotraining.cz`  
Stack: Next.js + Supabase  
Stav: interni MVP je funkcni a prakticky overeny v hlavnim toku.

## Co je hotovo

- Projekt je oddeleny od `vankotraining.cz` a `app.vankotraining.cz`.
- Supabase je napojena a aplikace cte realna data.
- Prihlaseni pres Supabase magic link funguje.
- Seznam aktivnich klientu se zobrazuje v hlavnim dashboardu.
- Detail klienta ukazuje profilova data a historii knee extension testu.
- Pridani noveho klienta funguje.
- Pridani noveho mereni funguje.
- Editace probehleho mereni funguje pro datum, vahu, delku berce a silu leve/prave nohy.
- Graf je zjednoduseny na spojnicovy graf s levou nohou, pravou nohou, asymetrii a normou.
- V grafu lze schovat/zobrazit levou nohu, pravou nohu a asymetrii.
- Archivace mereni funguje.
- Obnova archivovaneho mereni funguje.
- Archivace celeho klienta funguje.
- Obnova archivovaneho klienta funguje.
- Panel `Archiv klientu` je na webu a byl prakticky overeny.
- Normalni seznam klientu ukazuje jen aktivni klienty.
- Tlacitko archivace klienta je dole a jeho pozice je schvalena.
- Mobilni zobrazeni bylo upravene pro hlavni pouziti v telefonu.

## Supabase napojeni

Aplikace pouziva tyto tabulky pres Supabase client:

- `athletes`
- `athlete_profiles`
- `knee_extension_tests`

Aplikace pouziva tyto RPC funkce:

- `list_archived_athletes` - bezpecny vypis archivovanych klientu.
- `restore_athlete` - obnova archivovaneho klienta. Aplikace zkousi parametr `p_athlete_id`, pri starsi signature fallback `p_id`.
- `soft_delete_athlete` - archivace klienta. Aplikace zkousi `p_athlete_id` + `p_reason`, pri starsi signature fallback `p_id`.
- `list_archived_knee_extension_tests` - vypis archivovanych mereni pro vybraneho klienta.
- `restore_knee_extension_test` - obnova archivovaneho knee extension testu.

Poznamka: pridani/uprava aktivnich dat je zatim resena primo pres tabulky. Archivacni/obnovovaci workflow se opira o Supabase RPC a databazovou logiku.

## Rizika

- Aplikace stale obsahuje fallback na legacy anon key v `src/lib/supabase-browser.ts`. Anon key je verejny typ klice, ale cistsi dlouhodoby stav je spolehat se na Vercel env promennou a fallback pozdeji odstranit.
- Archivace mereni v UI historicky vychazi z akce mazani a aplikacni ochrany ji preznacuje na archivaci. Je nutne drzet Supabase trigger/RLS/RPC logiku v souladu, aby se z mereni nestalo skutecne nevratne mazani.
- Komponenty `ArchivedMeasurements`, `ClientDeletion` a `ButtonGuards` cte vybraneho klienta pres DOM/select a MutationObserver. Funguje to, ale je to krehci nez sdilena React state architektura.
- Obnova klienta reloaduje stranku. Pro MVP je to v poradku, ale pozdeji by bylo lepsi aktualizovat stav bez reloadu.
- Projekt nema automatizovane E2E testy. Smoke-test je zatim rucni.
- Supabase SQL migrace nejsou v tomto checkpointu kompletne auditovane proti produkcni databazi; bereme jako zdroj pravdy prakticke overeni na webu.

## Smoke-test checklist

Pred vetsim dalsim krokem projit na `knee.vankotraining.cz`:

1. Prihlasit se magic linkem.
2. Overit, ze seznam klientu ukazuje aktivni klienty a neukazuje archivovane klienty.
3. Vyhledat existujiciho klienta.
4. Otevrit detail klienta a zkontrolovat posledni test.
5. Prepnout mobilni zalozky `Mereni`, `Porovnani`, `Klient`.
6. Pridat testovaciho klienta.
7. Pridat testovaci knee extension mereni.
8. Upravit datum, vahu, delku berce a hodnoty sil u vytvoreneho mereni.
9. Archivovat vytvorene mereni.
10. Otevrit archiv mereni a obnovit mereni.
11. Archivovat testovaciho klienta.
12. Otevrit `Archiv klientu` a overit, ze klient je v archivu.
13. Obnovit klienta.
14. Overit, ze se klient vratil do aktivniho seznamu.
15. Zkontrolovat na mobilu, ze hlavni akce nejsou prekryte a tlacitko archivace klienta zustava dole.

## Stav dokonceni interniho MVP

Odhad: 85 % hotovo.

Duvod: hlavni interni workflow uz funguje od prihlaseni pres klienty, mereni, graf, archivaci a obnovu. Zbyvajicich 15 % tvori hlavne technicka stabilizace, cistsi state management, odstraneni provizornich fallbacku a automatizovany smoke-test.

## Backlog

1. Vyresit technicky dluh kolem DOM-based komunikace mezi komponentami a sjednotit vybraneho klienta pres React state nebo hlavni dashboard.
2. Zkontrolovat a pripadne ulozit finalni Supabase SQL migrace do repa jako dokumentovanou historii.
3. Odstranit legacy fallback anon key po potvrzeni, ze Vercel env promenne jsou stabilni.
4. Pridat minimalni automatizovany smoke-test pro prihlaseny stav nebo aspon test vypoctu Nm/kg/asymetrie.
5. Zlepsit stavove hlasky po archivaci/obnove bez nutnosti reloadu stranky.
6. Pridat jednoduchy export nebo tisk detailu klienta, az budou datove toky stabilni.
7. Navrhnout dalsi klinicky krok: interpretacni karta klienta, ktera shrne normu, deficit kg, asymetrii a prakticke doporuceni.

## Dalsi nejlepsi krok

Dalsi nejlepsi krok po stabilizaci: udelat malou technickou stabilizaci vyberu klienta a archivacnich panelu. Konkretne presunout `selectedAthlete` state do sdilene vrstvy v hlavnim dashboardu nebo sloucit archivacni panely tak, aby uz necetly vyber klienta z DOM. Tim se snizi riziko rozbiti pri dalsich UI upravach, hlavne na mobilu.
